# DuiJie 修改、推送 GitHub 与服务器部署标准流程

## 一、目的

这份文档用于统一 DuiJie 项目的日常发布流程，避免出现“本地已经改好，但 GitHub 没推、服务器没部署、数据库没迁移、PM2 没重启”的情况。

适用范围：

- Web 前端修改
- Node.js 后端修改
- 数据库迁移
- 文档更新
- 线上问题修复
- APK 相关后端版本接口更新

---

## 三、一次完整修改的标准顺序

### 1. 先确认修改范围

在开始改代码前，先确认本次修改属于哪一类：

- **仅前端**：页面/UI/交互/请求逻辑
- **仅后端**：控制器/服务/仓储/鉴权/接口逻辑
- **前后端联动**：接口字段、登录态、权限、页面联调
- **涉及数据库**：新增字段、补迁移、修历史数据
- **涉及部署**：Nginx、PM2、服务路径、构建产物

如果涉及数据库结构变化，必须同步准备 `migrations/*.sql`。

---

### 2. 修改代码

按实际需求修改对应代码。

常见位置：

- **前端**：`frontend/duijieReact/src`
- **后端**：`server/duijie/atomic`
- **迁移**：`server/duijie/migrations`
- **脚本**：`server/duijie/scripts`

原则：

- 只改与当前任务直接相关的文件
- 涉及公共能力时，要检查调用方是否需要一起调整
- 涉及鉴权、缓存、路由、会话恢复时，要检查前后端链路是否一致

---

### 3. 更新文档

#### 3.1 必须优先考虑更新 `CHANGELOG.md` 的情况

出现以下任意情况时，应更新 `CHANGELOG.md`：

- **用户可感知功能变更**
- **线上 Bug 修复**
- **登录、权限、刷新恢复、接口行为变化**
- **部署流程变化**
- **数据库迁移新增**
- **版本接口/APK 发版变更**

#### 3.2 需要更新 `PROJECT.md` / `ARCHITECTURE_GUIDE.md` 的情况

出现以下情况时，应同步更新架构文档：

- 新增模块或核心能力
- 权限模型变化
- 部署路径、启动方式、PM2、Nginx、迁移流程变化
- 新增长期维护规范

#### 3.3 文档更新原则

- `CHANGELOG.md`：写“这次改了什么”
- `PROJECT.md`：写“系统现在是怎么工作的”
- 独立专题文档：写“某件事如何执行”

---

### 4. 本地验证

修改后必须做最小充分验证。

#### 4.1 后端改动

在 `server/duijie` 下执行：

```bash
npm test -- tests/auth.test.js
```

如果改的是其他模块，应跑对应测试文件，例如：

```bash
npm test -- tests/client.test.js
npm test -- tests/project.test.js
npm test -- tests/ticket.test.js
```

#### 4.2 前端改动

在 `frontend/duijieReact` 下执行：

```bash
npm run build
```

如果是登录态、接口调用、页面刷新类问题，还要手工验证：

- 登录是否成功
- 刷新页面后是否仍保持登录
- 浏览器 Network 中 `/api/auth/me` 是否返回 `200`
- 是否出现旧缓存导致页面没更新

#### 4.3 涉及数据库迁移

在 `server/duijie` 下确认迁移文件存在，并执行：

```bash
npm run migrate
```

项目迁移运行器文件：

- `server/duijie/scripts/migrate.js`

迁移目录：

- `server/duijie/migrations`

---

### 5. 检查 Git 工作区

在仓库根目录执行：

```bash
git status -sb
```

发布前必须确认：

- 只包含本次任务相关文件
- 不把无关改动一起提交
- 新增迁移文件没有遗漏
- 文档修改已经包含在提交中

如果当前仓库同时存在很多其他未提交内容，必须先整理清楚再提交。

---

### 6. 提交到 GitHub

#### 6.1 提交前建议

先确认当前分支：

```bash
git branch --show-current
```

再查看远端：

```bash
git remote -v
```

#### 6.2 提交流程

```bash
git add <本次修改相关文件>
git commit -m "fix(auth): 修复刷新后掉登录"
git push origin main
```

注意：

- **推送到 GitHub 只是代码进入远端仓库**
- **并不等于线上服务器已经生效**

这是最容易漏掉的一步。

---

## 四、推送 GitHub 之后，还必须部署到服务器

DuiJie 当前不是自动 CI/CD 发布模式。

也就是说：

- 本地改完代码 ≠ 线上生效
- `git push origin main` ≠ 线上生效
- **只有服务器完成拉取、构建、迁移、重启后，线上才真正更新**

---

## 五、服务器手工部署标准流程

### 1. 登录服务器

```bash
ssh root@160.202.253.143
```

### 2. 拉取最新代码

```bash
cd /opt/duijie && git pull origin main
```

### 3. 安装后端依赖

```bash
cd /opt/duijie/server/duijie && npm install --production
```

### 4. 执行数据库迁移

```bash
cd /opt/duijie/server/duijie && npm run migrate
```

### 5. 重建前端（有前端改动时必须执行）

```bash
cd /opt/duijie/frontend/duijieReact && npm install && npx vite build
```

### 6. 重启后端服务

```bash
pm2 restart duijie
```

### 7. 查看日志确认启动情况

```bash
pm2 logs duijie
```

建议至少看这几类信息：

- 服务是否正常启动
- 是否有数据库报错
- 是否有迁移失败
- 是否有 `auth/me` 相关报错
- 是否有模块缺失、依赖缺失

## 六、可选：使用部署脚本发布

仓库根目录存在：

- `deploy_update.py`

它的作用包括：

- 上传后端代码到服务器
- 安装后端依赖
- 上传迁移文件并执行 `scripts/migrate.js`
- 上传前端 `dist`
- `pm2 restart duijie`

### 使用前提

- 你本机可 SSH 登录服务器
- 知道服务器密码，或者环境变量里有 `SSH_PASS`
- 本地前端已经先构建好 `dist`

### 典型使用顺序

#### 1. 先本地构建前端

```bash
cd frontend/duijieReact && npm run build
```

#### 2. 再执行部署脚本

```bash
python deploy_update.py
```

如果脚本未拿到密码，会提示输入服务器密码。

### 适用场景

- 需要快速同步当前本地代码到服务器
- 不想在服务器上手工逐条执行上传/迁移/重启命令

### 注意

- 脚本依赖你本地当前工作区内容
- 如果本地还有未整理的改动，也会一起影响部署结果
- 所以在执行脚本前，仍建议先确认 `git status -sb`

## 七、部署完成后的验证清单

### 1. 页面验证

- 访问线上首页：`http://160.202.253.143:8080`
- 检查页面是否能正常打开
- 强刷浏览器：`Ctrl + F5`

### 2. 登录态验证

如果本次改动涉及登录、鉴权、刷新恢复：

- 登录账号
- 刷新页面
- 打开浏览器 `Network`
- 检查 `/api/auth/me`

结果判断：

- **200**：登录恢复成功
- **401**：未带 token/cookie，属于未登录/鉴权失败
- **500**：后端接口内部报错，需要查服务器日志

### 3. PM2 与日志验证

```bash
pm2 show duijie
pm2 logs duijie --lines 100
```

### 4. 如果改了 APK 版本接口

还要额外检查：

- `http://160.202.253.143:8080/api/app/version`
- `http://160.202.253.143:8080/downloads/duijie.apk`

## 八、这套流程里最容易漏掉的点

### 1. 只改了本地，没有推 GitHub

表现：

- 本地文件已改
- 远端仓库还是旧代码
- 服务器 `git pull` 也拉不到新内容

### 2. 推了 GitHub，但没部署服务器

表现：

- GitHub 已更新
- 线上页面、接口仍然是旧版本

### 3. 部署了后端，但没跑迁移

表现：

- 新代码依赖新字段/新表
- 线上接口报 SQL 错误
- 常见结果是 `500`

### 4. 部署了后端，但没重建前端

表现：

- 后端已是新逻辑
- 浏览器仍在加载旧前端 bundle
- 页面看起来“像没改”

### 5. 重建或拉代码后没重启 PM2

表现：

- 服务器磁盘上的代码是新的
- 实际运行进程还是旧代码

### 6. 浏览器缓存旧前端资源

表现：

- 服务器已更新
- 页面仍显示旧逻辑或旧文案

处理方式：

- `Ctrl + F5` 强刷
- 必要时清除站点缓存

## 九、DuiJie 的推荐发布闭环

以后每次涉及正式修改，建议固定按下面顺序执行：

1. **确认需求与影响范围**
2. **修改代码**
3. **补齐迁移文件（如需要）**
4. **更新 `CHANGELOG.md`（如属于发布级改动）**
5. **更新 `PROJECT.md` / 专题文档（如属于架构或流程变化）**
6. **本地测试与构建**
7. **检查 `git status -sb`，确保只包含本次变更**
8. **提交并 push 到 GitHub**
9. **部署服务器：pull / install / migrate / build / pm2 restart**
10. **浏览器与日志双重验证**

## 十一、针对这次“刷新后掉登录”的特别说明

这类问题的标准排查顺序应固定为：

1. 登录成功后刷新页面
2. 打开浏览器 Network
3. 检查 `/api/auth/me` 状态码
4. 若为 `500`，立刻查 `pm2 logs duijie`
5. 核对服务器是否：
   - 已拉取最新代码
   - 已执行迁移
   - 已重启 PM2
   - 已重建前端
6. 必要时强刷浏览器缓存

只要这条链路固定下来，后续类似问题会定位得很快。
