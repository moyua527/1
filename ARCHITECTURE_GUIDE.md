## 一、系统定位

DuiJie 是一套以 **企业身份 + 项目协作** 为中心的客户项目对接平台。

系统的核心业务链路如下：

```text
用户登录
  -> 确定当前用户身份
  -> 确定当前活跃企业
  -> 计算平台权限 / 企业权限 / 项目权限
  -> 进入项目、客户、任务等业务页面
  -> 发起前端 API 请求
  -> 后端路由 / 中间件 / 控制器 / 服务 / 仓储处理
  -> MySQL 持久化
  -> 返回结果并在部分场景触发实时同步
```

从业务上看，系统最核心的是：

- 认证与用户
- 企业与组织身份
- 项目管理
- 任务推进
- 消息协作

客户、商机、文件、工单、报表则围绕主链补全业务闭环。

## 四、模块主次关系

## 4.1 一级核心模块

这些模块决定系统是否成立，是主干中的主干。

### 1）认证与用户模块

作用：

- 登录
- 注册
- 验证码登录
- 重置密码
- 2FA
- 获取当前用户信息

为什么重要：

- 它决定“你是谁”
- 所有业务都依赖它完成身份识别

### 2）企业模块

作用：

- 当前活跃企业管理
- 企业成员管理
- 企业角色管理
- 企业部门管理
- 企业切换

为什么重要：

- 它决定“你当前以哪个企业视角工作”
- 直接影响项目成员展示、企业权限与视角切换

### 3）权限模块

作用：

- 平台角色权限（当前一级身份只有 **admin** 和 **member** 两种）
- 企业权限（企业内自定义角色与权限字段）
- 项目权限（企业角色在项目中的绑定结果）

为什么重要：

- 平台角色决定全局菜单可见性：admin 拥有全部功能，member 仅拥有基本业务功能
- 企业权限和项目权限在此基础上叠加，决定具体的编辑、删除、管理能力
- 页面入口、按钮、编辑能力都由三层权限共同控制

### 4）项目模块

作用：

- 项目列表
- 项目详情
- 项目成员
- 项目企业关系
- 项目应用链接
- 项目内任务、消息、里程碑入口

为什么重要：

- 它是整个平台业务中心
- 客户、企业、任务、消息等都围绕项目汇聚

### 5）任务模块

作用：

- 全局任务看板
- 项目内任务执行
- 状态流转
- 优先级与附件

为什么重要：

- 它是项目落地执行层
- 项目是否推进，主要看任务模块

## 五、前后端连接方式

## 5.1 标准调用链

```text
前端页面
  -> services/api
  -> /api/xxx
  -> route
  -> middleware
  -> controller
  -> service
  -> repository
  -> db
  -> res.json
  -> 前端状态更新
```

## 5.2 权限链

```text
平台角色权限（stores/permissions.tsx）
  当前一级身份只有 admin 和 member
  admin = 全部功能
  member = 基本业务功能
  + 企业权限（useUserStore.enterprisePerms）
    企业内自定义角色决定：创建项目、管理客户、查看报表、管理任务等
  + 项目权限（useProjectPerms）
    项目中的企业角色绑定结果决定：编辑项目、删除项目、管理成员、管理任务等
  = 页面显示权限 + 按钮权限 + 操作权限
```

> 注：`permissions.tsx` 现在只保留 admin / member 两个平台基线权限；企业权限与项目权限分别在企业层和项目层叠加生效。

## 5.3 实时链

```text
后端写操作
  -> 广播 data_changed / 消息事件
  -> 前端 socket 监听
  -> useLiveData / 页面刷新
```

## 十、总结

DuiJie 不是由多个孤立页面组成，而是一个围绕 **企业身份与项目协作** 组织起来的平台。

可以用一句话概括：

```text
认证决定你是谁
企业决定你站在哪个组织视角
权限决定你能做什么
项目决定你当前处理什么工作对象
任务 / 消息 / 里程碑决定工作如何推进
客户 / 文件 / 工单 / 报表补全业务闭环
```

因此，理解本系统最重要的不是单看某个页面，而是先抓住这条主链：

- 认证
- 企业
- 权限
- 项目
- 任务/协同

这条主链理顺后，其它模块都能顺着挂接理解。
